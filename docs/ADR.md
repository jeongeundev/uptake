# Architecture Decision Records

## 철학
**검증 가능한 재사용.** uptake는 방법론을 옮겨 심되, 옮겨졌음을 실제 검증(green + red)으로 증명한다. 태도는 **서술적**이다 — "성공 repo가 *실제로* 이렇게 한다 + 트레이드오프"이지 "이게 정답"이 아니다. **정직**을 최우선으로 둔다 — 성공 위장 금지, provenance 필수, 실패는 표면화. 구현은 **최소**로(요청하지 않은 유연성·미래대비 금지). uptake는 자기가 설파하는 방법론을 **dogfooding**한다.

---

### ADR-001: 재사용 대상 = 코드가 아니라 개발 방법론
**결정**: 오픈소스 재사용의 "3층"(코드 → 패키지 → **프로세스 지식**) 중 3층을 연다. 타깃은 방법론이 파일로 물질화된 **AI-native repo**(`.claude/`, `.specify/`, CI 등).
**이유**: 1·2층은 패키지 매니저가 열었지만 3층은 여전히 손으로 복사·역설계된다. AI-native repo에서 방법론은 이미 파일이므로 기계적으로 다룰 수 있다.
**트레이드오프**: AI-native가 아닌 repo는 범위 밖. 방법론이 암묵지로만 있는 곳은 다루지 못한다.

### ADR-002: MVP 앵커 = Spec↔Verification 루프 하나
**결정**: MVP는 단일 패턴 — Spec↔Verification 루프 — 하나에만 전체 파이프라인을 건다.
**이유**: 강한 성공 기준(눈에 보이는 green/red 데모)을 하나 확보하는 것이 폭넓은 얕은 커버리지보다 증명력이 크다. 이 루프는 게이트형의 원형이라 판별 오라클이 자연스럽다.
**트레이드오프**: 다양성 데모를 포기. "하나만 되네"라는 인상 리스크는 카탈로그의 서술 전용(`descriptive`) 패턴으로 완화.

### ADR-003: 범위 원칙 = 탐지는 넓게 / 생성은 깊게
**결정**: 카탈로그(서술)는 넓게, 생성·검증은 앵커 루프에만 깊게.
**이유**: 자원을 깊이에 몰아 증명 가능한 한 갈래를 완성. 넓이는 서술로 커버.
**트레이드오프**: "넓게"에 해당하는 자동 채굴은 MVP에서 비전으로 남는다(ADR 참조: 손 큐레이션).

### ADR-004: 3단계 번역 엔진 — 환원 불가능한 가치 = 추상화(떼어내기)
**결정**: EXTRACT→ABSTRACT→INSTANTIATE. **핵심 가치는 ABSTRACT** — 원본 스택에서 본질을 떼어내는 것. 발견(A)은 입력, 이식·검증(C)은 증명.
**이유**: 복사·fork가 근본적으로 못 하는 유일한 것이 "원본과의 결합을 끊는 것"이다. 여기에만 해자가 있다.
**트레이드오프**: 추상화가 어려운 패턴은 다루기 힘들다. 발견·이식만으론 차별화가 얕아 이 순서를 고정한다.

### ADR-005: 증거 기반 = 다중 예시(대조) 추상화 + 신뢰도 생애주기
**결정**: 본질/파라미터 분리는 **≥2개 repo의 대조**(공통=본질 / 차이=파라미터)로 한다. 이 축의 필드명은 **`evidenceStatus`**: `observed`(N=1, tentative) → `corroborated`(N≥2). (ADR-012의 `capability` 축과 직교한다 — 둘 다 "tier"로 부르지 않는다.)
**이유**: 단일 예시로는 우연과 본질을 구별할 수 없어 환각 위험이 크다. 대조는 falsifiable 기준을 문자 그대로 실행하고, "카탈로그가 클수록 추상화가 날카로워진다"는 플라이휠에 기계적 근거를 준다.
**트레이드오프**: 콜드스타트에 씨앗 클러스터가 필요. 씨앗의 우연한 공통점을 본질로 오인할 수 있음(→ 다양한 스택의 씨앗으로 완화, M0에서 검증).

**보완 — 씨앗의 독립성 (N을 세는 규칙)**: `N≥2`만으로는 서로를 복사한 두 repo가 독립 증거로 계산된다. 그래서 다음을 만족해야 **독립 사례 2건**으로 센다.
- **fork 관계**이거나 **동일 스캐폴딩 템플릿**에서 파생된 repo는 **하나로 센다**.
- **동일 조직·동일 메인테이너**의 repo는 독립으로 세지 않는다 — 관습은 조직 단위로 전파되므로 공통점이 본질이 아니라 사내 규약일 수 있다.
- 씨앗 중 **최소 하나는 타깃과 다른 스택**이어야 한다(ADR-013). 스택 다양성은 "복사가 아님"의 가장 강한 기계적 증거다.
- **판정은 사람(큐레이터)이 한다.** LLM은 후보를 제시할 뿐이다. 독립성 판정은 패턴 파일에 `sources[].independenceGroup`(같은 값 = 1건으로 카운트)과 `independenceNote`(왜 그렇게 판정했는지)로 직렬화되며, 엔진은 그 값을 **검사만** 한다 — GitHub 메타데이터로 자동 추론하지 않는다(로컬-우선·의존성 0 범위 밖이고, 판정 책임이 LLM으로 넘어가면 서술적 태도가 무너진다).
- **"공통점이 본질"이라는 판정** 역시 큐레이터의 몫이다. 어느 파일에서 무엇을 관찰했는지는 `provenance[].observedRole`로 남고, 근거 없이는 승급하지 않는다.
- **생존자 편향은 라벨한다** — 관찰 대상이 성공한 repo라는 사실(ADR-006)은 패턴 파일의 `tradeoffs`에 적히고 카탈로그 UI에 함께 표시된다. 숨기지 않는다.
- **N을 세는 단위는 "근거 파일이 달린 repo"다.** 실재하는 provenance가 하나도 연결되지 않은 repo는 세지 않으며, 그런 고아 항목이 있으면 패턴 자체가 거부된다. repo 이름을 하나 더 적어서 승급하는 경로는 없다 — 이것이 "근거 없이는 승급하지 않는다"의 기계적 형태다.
- **미달 시 로드 거부.** `corroborated`라고 선언했는데 위 조건을 만족하지 못하면 카탈로그에서 거부한다. 엔진이 `observed`로 조용히 강등하지 않는다 — 큐레이터의 선언과 데이터의 불일치는 고쳐야 할 오류이지 자동 보정할 상태가 아니다.

### ADR-006: 태도 = 서술적(비규범적)
**결정**: uptake는 "성공 repo가 실제로 이렇게 한다 + 트레이드오프"만 말한다. "이게 정답"이라는 규범적 단정 금지.
**이유**: 본질은 관찰된 공통점이어야지 LLM이 지어낸 주장이면 안 된다. 규범적 태도는 환각·과신으로 이어진다.
**트레이드오프**: "그래서 뭘 써야 하냐"는 직접 처방을 원하는 사용자에겐 덜 친절. 또한 관찰 대상이 성공한 repo(=생존자)라 그 관습이 성공의 *원인*이라는 보장은 없다 — 이 **생존자 편향**은 숨기지 않고 정직하게 라벨한다.

### ADR-007: 생성 게이트 = `corroborated`(N≥2)
**결정**: 남의 repo에 써 넣는 이식·생성은 `corroborated` 패턴에서만 점등. `observed`(N=1)는 등재는 하되 생성은 experimental 옵트인으로 MVP에서 지연.
**이유**: 한 번 본 것을 "재사용 가능"이라 주장하며 코드에 심는 것은 ADR-006·009와 충돌한다. 카탈로그화(관찰 서술)와 생성(재사용 주장)은 다른 층위다.
**트레이드오프**: "아무 repo나 넣으면 마법처럼" 되는 데모 wow를 일부 포기.

### ADR-008: 자기검증 = 양성 + 음성(판별 오라클)
**결정**: VERIFY는 "green 실행"이 아니라 **판별 행동 시연**이다 — 준수 상태 green **그리고** 심은 위반을 red로 잡음. 패턴 스키마에 **판별 오라클**을 넣는다. 성공 위장 절대 금지, 실패는 정직하게 표면화.
**이유**: 아무것도 안 하는 게이트도 green이다. 판별력(나쁜 입력에서 실패)만이 "루프를 닫았음"을 증명한다. 위반 케이스는 패턴 정의에서 파생해 자기채점(tautology)을 막는다.
**트레이드오프**: 패턴마다 오라클 정의라는 추가 부담. 그러나 이것이 없으면 "검증"이 연극이 되므로 load-bearing.

### ADR-009: 검증 가능한 provenance 필수
**결정**: 모든 추출·생성 결과는 resolve 가능한 실재 소스 파일 경로를 달아야 한다. resolve 안 되면 폐기. 환각 금지.
**이유**: 서술적 태도와 OSS 출처 윤리의 물질적 근거. provenance 없는 주장은 검증 불가능한 환각. 실재 소스 경로 강제는 **인젝션의 자연 필터**이기도 하다(불신 격리와 상호보완) — 지어낸 지시는 resolve되는 출처가 없어 걸러진다.
**트레이드오프**: 소스 경로가 사라지거나 불명확한 방법론은 카탈로그에 못 올린다.

### ADR-010: 주 사용자 = 사람 드라이버 / repo-native 산출물 / human-in-the-loop
**결정**: 사람이 패턴을 고르고 diff를 승인한다. 이식 결과물은 repo-native 산출물(스펙 게이트·hook·CLAUDE.md 규칙)이다. 에이전트가 그 방법론을 따르게 되는 것은 **부수효과**.
**이유**: diff-미리보기·명시적 적용이라는 안전 서사와 대회 legibility(눈에 보이는 데모)를 위해 사람이 루프 안에 있어야 한다. 에이전트 소비의 가치는 repo-native 산출물로 공짜로 따라온다.
**트레이드오프**: 완전 자율 에이전트 통합의 우아함은 MVP에서 포기.

### ADR-011: 커먼즈 = 포터블 파일 포맷 + 로컬 카탈로그
**결정**: MVP의 "커먼즈(영혼)"는 **포터블한 패턴 파일 + 로컬 카탈로그(+ export)** 로 실재한다. 호스팅 레지스트리·기여 인프라·승급 자동화는 MVP 밖.
**이유**: 영혼은 서비스가 아니라 "패턴이 원본에서 떼어져 파일 하나로 옮겨 다닌다"는 사실 자체다. 그건 5-구성요소 스키마에 이미 내장돼 있다. 플라이휠 회전은 사용자가 여럿일 때 성립하므로 데모에선 서사로 둔다.
**export의 실체**: 패턴이 자립적인 JSON 파일 하나라는 사실 **자체가** export다 — `catalog/<patternId>.json`을 복사하면 옮겨진다. 별도의 export 기능(번들링·전용 UI·왕복 도구)은 MVP 밖이고 수용 기준도 두지 않는다. 포맷 선택(JSON)이 이 성질을 이미 보장한다.
**트레이드오프**: "자동으로 자라는 카탈로그"는 화면이 아니라 말로 보여준다.

### ADR-012: 카탈로그 `capability` 2분류 — 생성형(게이트형) / 서술 전용(비게이트형)
**결정**: 카탈로그는 넓게(모든 관찰 방법론 서술), 생성 파이프라인은 좁게(판별 오라클이 있는 게이트형만). 비게이트형 방법론(리뷰 관습·문서 습관 등)은 서술 전용. 이 축의 필드명은 **`capability`**: `generative`(게이트형) / `descriptive`(비게이트형). ADR-005의 `evidenceStatus`와 **직교**하며, 생성 점등은 두 축을 모두 만족할 때만 일어난다(`generative` **AND** `corroborated`).
**이유**: 생성+검증 기계는 게이트형에만 근본적으로 적용된다. "모든 방법론을 자동 이식"이라는 과대주장을 차단하고, 서술적 태도에 자연스러운 자리를 준다.
**트레이드오프**: 개발 문화의 상당 부분은 uptake에서 영원히 서술 전용 — 옮겨 심지 못한다.

### ADR-013: 타깃 스택 = JS/TS(vitest) 하나 / 소스 ≠ 타깃 필수
**결정**: MVP 이식 타깃은 JS/TS(vitest) 한 스택. 씨앗 클러스터엔 타깃과 **다른 스택** repo를 최소 하나 포함한다.
**이유**: 게이트 생성·결합점 탐지는 스택-종속이라 하나로 한정해야 크기가 통제된다. 소스=타깃이면 "이름만 바꿔 복사"와 구별 불가 — 소스≠타깃이어야 B(떼어내기)가 데모에서 실제로 증명된다.
**트레이드오프**: 다중 타깃 스택 지원은 이후로. 씨앗 큐레이션에 스택 다양성 제약이 붙는다.
