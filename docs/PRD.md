# PRD: uptake

## 목표
오픈소스 저장소에 **결합된 채로 존재하는 개발 방법론**을 재사용 가능한 **파라미터화된 패턴**으로 추상화해, 다른 프로젝트에 **검증과 함께** 이식한다 — "오픈소스가 코드에 해준 것을 개발 방법론에 한다."

## 사용자
검증된 오픈소스가 가진 개발 규율(예: Spec↔Verification 루프)을, 손으로 역설계하지 않고 자기 프로젝트에 이식하고 싶은 **개발자 / 소규모 팀**.

- 사용자는 **드라이버**다 — 패턴을 고르고, 생성된 diff를 검토·승인한다 (human-in-the-loop).
- 이식 결과물의 **소비자**는 타깃 repo의 개발 루프 전체 — 사람 **그리고** 그 repo에서 일하는 코딩 에이전트. 에이전트가 자동으로 방법론을 따르게 되는 것은 repo-native 산출물의 **부수효과**이지, 별도의 주 사용자가 아니다.

## 핵심 기능
uptake의 세 기능은 동등하지 않다. **발견(입력) → 추상화(핵심 가치) → 이식·검증(증명)** 의 관계다.

1. **발견 / 카탈로그 (입력)**
   손으로 큐레이션한 씨앗 클러스터에서 패턴을 카탈로그로 제공하며, `capability` 축으로 2분류한다.
   - *생성형(`generative`, 게이트형)*: 통과/실패를 판별할 수 있는 패턴 → 전체 파이프라인 대상
   - *서술 전용(`descriptive`, 비게이트형)*: 판별 오라클이 없는 방법론 → 등재·서술만
   각 패턴은 5-구성요소 스키마로 기술된다: **의도 · 불변 구조(역할) · 결합점(파라미터) · 출처(provenance) · 판별 오라클**.

2. **추상화 (핵심 가치 — 프로젝트의 영혼)**
   **≥2개 실재 repo를 대조**해 스택-불변 **본질**(의도 + 불변 구조)과 스택-종속 **결합점**(파라미터)을 분리한다(EXTRACT→ABSTRACT). 복사가 근본적으로 못 하는 유일한 것 — "원본 스택에서 떼어내기" — 을 한다. 단일 관찰(N=1)은 **tentative(`observed`)** 로만 등재하며, 대조로 검증되면 **`corroborated`(N≥2)** 로 승급한다.

3. **이식 + 검증 (증명)**
   타깃 repo의 **결합점을 자동 탐지**해 패턴을 재생성하고(INSTANTIATE — 생성물엔 **왜(근거) 주석 + provenance**를 단다), 두 방향으로 실제 검증한다(VERIFY):
   - **양성**: 준수 상태 → 게이트 green
   - **음성**: 심어놓은 스펙 위반 → 게이트가 red로 **잡아냄**
   음성 테스트가 물어야만 추상화가 성립했음이 증명된다. **diff 미리보기 → 명시적 승인** 후에만 타깃 repo에 적용한다.

## MVP 수용 기준
이 프로젝트는 자기가 설파하는 방법론을 dogfooding하므로, PRD 자체가 **테스트 가능한 스펙**이어야 한다. 아래 기준은 구현 전에 그대로 테스트로 옮겨진다. 하나라도 red면 MVP는 완성이 아니다.

| ID | 기준 | 검증 |
|---|---|---|
| AC-1 | 각 `corroborated` 패턴은 서로 다른 `independenceGroup`을 가진 `sources` **≥2개**를 갖고, 그중 최소 하나가 `isTargetStack: false`다. 세는 대상은 **근거 파일(provenance)이 연결된 source뿐**이다. **못 만족하면 로드 거부**(선언과 근거의 불일치 = 데이터 오류, 조용한 `observed` 강등 없음) | `independenceGroup` distinct <2 fixture · 전부 `isTargetStack: true`인 fixture 각각이 로드 거부되는지 검사. 판정값 자체는 큐레이터가 부여한다(ADR-005) |
| AC-2 | **로드 거부(하드 게이트)**: provenance resolve 실패, `sources`·`provenance` 빈 배열, 참조 무결성 위반(**양방향** — 가리키는 대상 없음 / **고아 source** / **고아 role**), `capability`↔`oracle` **양방향** 불일치, `observed`인데 독립 그룹이 1이 아님 | 거부 사유별 fixture 각각에 대해 로드 거부 검사. **더미-source fixture**(근거 파일 없는 source로 `independenceGroup`만 채워 `corroborated`를 노리는 케이스), **고아 role fixture**(근거 없이 선언된 역할), `observed`인데 독립 그룹 0개·2개인 fixture 포함 |
| AC-2b | **역할 단위 대조**: `corroborated` 패턴의 **각 role**은 서로 다른 `independenceGroup`의 provenance ≥2개로 뒷받침된다. 한 repo에서만 관찰된 것은 `roles`(본질)가 아니라 `bindingPoints`(차이)에 속한다 | role별 독립 그룹 수 검사. **role A는 repo1에서만, role B는 repo2에서만 관찰된 fixture**가 로드 거부되는지 — 패턴 전체 N≥2만으로는 대조 추상화가 성립하지 않음 |
| AC-3 | **생성 차단(소프트 게이트)**: 로드된 패턴 중 `observed`이거나 `descriptive`인 것은 카탈로그에 **보이되** 생성이 비활성이다 | `observed` / `descriptive` / 둘 다인 조합 각각을 UI·엔진 양쪽에서 차단 검사 |
| AC-4 | 타깃 repo에서 패턴의 각 결합점을 **근거 파일과 함께** 제시한다. 탐지 실패한 결합점은 `binding-unresolved`로 표시하고 사용자 입력을 받는다 | fixture repo에 대해 결합점별 값+근거 경로 검사, 미탐지 케이스 포함 |
| AC-5 | 생성·검증 전 구간에서 **타깃 repo는 수정되지 않는다** | 검증 전후 타깃 repo 작업 트리 동일성 검사 (git status + 파일 해시) |
| AC-6 | 양성: 생성물을 적용한 격리 워크스페이스에서 게이트가 **green** | 준수 fixture 실행 |
| AC-7 | 음성: 오라클이 정의한 위반을 심은 워크스페이스에서 게이트가 **red**로 잡는다. red의 정의는 exit code가 아니라 **리포터 출력상 `oracle.gateTestId` 테스트의 실패**다 | 위반 fixture 실행. 삽입 불가 fixture(경로 이탈, `marker` 0회·2회 이상)는 `injection-failed`로 거부되는지 함께 검사 |
| AC-7b | **인프라 오류는 음성 성공이 아니다** — 리포터를 산출하지 못한 실행(설치 실패, 설정·문법 오류, spawn 실패, signal, timeout)은 `gate-error`이며 이식이 중단된다 | 각 오류를 심은 fixture가 `gate-error`로 보고되는지 검사. **깨진 vitest 설정 fixture가 "위반을 잡았다"로 계산되지 않는 것**이 핵심 케이스 |
| AC-8 | 음성이 green이면 이식 전체가 **실패**로 표면화되고 적용이 차단된다. 이 실패는 일반 테스트 실패와 **구별되는 상태**(`negative-not-caught`)로 보고된다 | 판별력 없는 게이트(항상 green) fixture가 실패로 보고되는지 검사 |
| AC-9 | 사용자가 승인한 diff는 **검증에 사용된 산출물과 동일**하다. 불일치 시 적용을 거부한다 | 승인 직전 산출물 내용 해시 재확인 테스트 |
| AC-10 | 타깃 repo 쓰기는 **사용자 승인 후에만** 일어난다 | 승인 없는 경로에서 쓰기 호출이 없음을 검사 |
| AC-11 | 소스 스택 ≠ 타깃 스택인 이식 한 건이 고름→탐지→생성→양성/음성→승인→적용까지 끝까지 통과한다 | E2E 시나리오 1건 |
| AC-12 | 실행되는 게이트 커맨드는 이식 실행 개시 전 사용자에게 **표시된 것과 동일**하다. 표시되지 않은 커맨드는 실행하지 않는다 | 실행 래퍼가 사전 공개된 argv로만 호출되는지 검사 (검증 도중 별도 승인은 받지 않음 — `ARCHITECTURE.md` 신뢰 경계) |

## MVP 제외 사항
- **자동 소스측 패턴 채굴 (개방형 발견) = E2 비전** — 사용자가 저장소만 지정하면 앱이 방법론을 스스로 발견해 후보로 제안하는 개방형 발견은 E2로 미룬다. phase 2의 **대상 지정 추출**(사용자가 방법론을 함께 지정)은 이 개방형의 선행·축소 단계이며, 개방형은 그 위에 얹는다. MVP의 씨앗은 손 큐레이션이었다(M0 스파이크).
- **EXTRACT·ABSTRACT의 앱 구현 (MVP 한정 제외 — phase 2에서 해제)** — MVP에선 카탈로그 제작이 **오프라인 손 큐레이션 절차**였고 앱이 구현한 것은 INSTANTIATE·VERIFY뿐이었다. phase 2는 대상 지정 추출로 EXTRACT·ABSTRACT를 앱 안에 들인다(아래 'Phase 2 범위' 참조). 자동 채굴이 MVP 밖인 것(위)과 대상 지정 추출을 앱 기능으로 만드는 것은 별개다. 핵심 가치가 ABSTRACT라는 것(ADR-004)을 이제 앱이 실제로 수행한다.
- **N=1(`observed`) 패턴의 생성** — experimental 옵트인으로 지연. MVP는 `corroborated`만 생성.
- **비게이트형 방법론의 이식** — 서술 전용. 검증 불가한 것은 옮겨 심지 않는다.
- **"내 repo에 어떤 방법론이 빠졌는지 진단"하는 Discovery 시나리오** — MVP 이후 별도 트랙.
- **다중 패턴 워크플로우 조합** — MVP는 단일 패턴(Spec↔Verify) 하나만 깊게.
- **호스팅된 공유 레지스트리 / 기여 인프라 / 승급 자동화** — 커먼즈는 포터블 파일 포맷 + 로컬 카탈로그로만 실재.
- **다중 타깃 스택** — MVP 타깃은 JS/TS(vitest) 하나. (단, 소스 스택 ≠ 타깃 스택은 필수 — 아래 참조.)

## Phase 2 범위 — 카탈로그 저작 (EXTRACT·ABSTRACT)
MVP가 INSTANTIATE·VERIFY(이식·증명)를 앱으로 닫았다면, phase 2는 그 **앞단**을 앱으로 들인다 — 카탈로그를 손이 아니라 앱이 저작한다. 이번 phase의 목표는 **하드코딩 없는 범용 분석 파이프라인의 기반**을 세우는 것이다: 사용자가 소스 저장소 ≥2개와 의도를 주면, 앱이 provenance 기반으로 근거를 수집·대조해 패턴 초안을 제안하고, 사용자가 검토·승인한 뒤에만 카탈로그에 저장한다.

**범위의 정직한 경계.** "어떤 저장소에도"는 **분석 대상(소스) 측**을 뜻한다 — 이식 타깃은 ADR-013(JS/TS vitest) 그대로다. **분석(EXTRACT·ABSTRACT)과 `descriptive` 등재까지는 완전 범용**이고, **`generative`의 이식·검증은 현재 엔진이 합성하는 앵커 형태(3역할)에 한정**한다. 임의 패턴의 자동 합성·이식은 이번 범위 밖(후속 phase). 사용자가 저장소만 지정하면 앱이 방법론을 스스로 발견하는 **개방형 발견은 E2 비전** — 이번 데이터 모델·인터페이스는 그 확장을 막지 않되, 개방형을 위한 유연성을 지금 구현하지는 않는다(ADR-014).

**입력.** 사용자는 소스 저장소(`UPTAKE_SOURCE_ROOT` 아래)와 **추출할 방법론의 의도(intent)**만 지정한다. 앱(LLM proposer)이 role·binding·provenance 후보로 초안을 제안하고, 사용자는 **승인/거부만** 한다(후보 편집은 후속). `revision`은 저작 개시 시점 HEAD SHA로 자동 고정한다.

요구사항(착수 시 수용 기준으로 물질화):
- **대조 근거 ≥2 독립 저장소.** `corroborated` 패턴은 최소 2개의 독립 저장소가 있어야 생성 가능하다. 독립성·역할 단위 대조 판정은 기존 층 1 하드 게이트 기준을 그대로 쓴다(AC-1·AC-2b).
- **모든 근거에 파일 경로 + revision provenance.** 근거는 저작 개시 시 고정한 커밋 SHA에 결속되며, resolve되지 않는 주장은 초안에 담기지 않는다(ADR-009). resolve는 기존 루트 기반 계약을 그대로 재사용한다.
- **저장소 내용은 불신 데이터.** EXTRACT가 읽는 소스 저장소 내용은 지시가 아니라 데이터로 격리한다. 읽기 전용이며 실행하지 않는다(신뢰 경계).
- **공통점=role 후보 / 차이점=binding 후보.** 소스 간 대조로 공통점은 `roles`(본질) 후보로, 차이점은 `bindingPoints`(파라미터) 후보로 분리한다. **`corroborated` 저작에 한해** 단일 저장소(단일 `independenceGroup`)만 지지하는 role 후보는 role이 아니라 binding으로 내려간다(ADR-005).
- **판정은 사용자 몫.** `independenceGroup`·`isTargetStack`·"공통점이 본질"의 **최종 해석·승인은 사용자**가 한다. 앱은 provenance·target stack 탐지 결과 등 **결정적 사실과 그 근거를 자동으로 제시**할 수 있으나, 판정을 대신 내리지 않는다(ADR-005).
- **LLM은 후보만(ADR-015).** EXTRACT·ABSTRACT·oracle 초안은 proposer 포트를 통한 LLM 후보이고, 수용 기준 테스트는 이 포트를 **스텁**으로 주입해 결정적 기계만 검증한다.
- **환각 봉쇄(AC-C9).** proposer가 resolve 불가 근거·단일 repo 지지 role·판별력 없는 oracle을 제안하면 각각 등재가 거부되고 사유가 보고된다.
- **승인 전 catalog 미기록.** 초안은 승인 전까지 카탈로그에 쓰지 않는다. 승인 시에만 `catalog/<patternId>.json`으로 기록한다(타깃 쓰기의 AC-10과 대칭).
- **씨앗 보호(AC-C10).** 등재는 기존 `catalog/` 파일을 덮어쓰지 않는다. patternId 충돌 시 등재가 거부되고 기존 파일은 불변이다(기존 패턴 편집·교체는 후속).
- **승인된 패턴은 기존 하드 게이트 통과.** 등재되는 패턴은 층 1 하드 게이트(provenance resolve·참조 무결성·독립성/스택·`capability`↔`oracle` 일치)를 기록 전·후로 통과해야 한다.
- **oracle 자기검증.** generative 초안의 oracle은 앱이 초안하되, 등재 전에 **번들 정규 fixture 타깃**에 대해 실제 VERIFY(양성 green + 음성 red)를 돌려 판별력을 입증한 것만 등재한다(ADR-016). 앵커 형태에 한정한다.
- **전체 흐름 연결(앵커 형태).** 앱이 저작·등재한 앵커 형태 generative 패턴 하나가 곧바로 INSTANTIATE(역할 형태 매칭)→VERIFY로 이식·검증까지 이어진다(E2E 1건).

세부 설계는 `ARCHITECTURE.md`의 'EXTRACT·ABSTRACT 저작 계약', 결정 근거는 `ADR.md`의 ADR-014~016을 따른다.

## 데모의 필수조건 (대회 legibility)
MVP는 **소스 스택 ≠ 타깃 스택**으로 이식을 시연한다(예: PHP repo에서 관찰 → JS/TS repo에 이식). 소스와 타깃이 같은 스택이면 "복사"와 구별되지 않아 핵심 가치(B)를 증명하지 못한다. 데모는 하나의 `corroborated` 패턴을 **끝까지 깊게**(고름 → 결합점 탐지 → 이식 → 양성/음성 검증 → diff 승인) 보여준다.
